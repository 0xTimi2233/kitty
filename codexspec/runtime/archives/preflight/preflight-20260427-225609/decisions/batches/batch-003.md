# Decision Batch 003

本批只补充 Batch 002 Q3 的用户自定义决策。用户已决定“开发代码时约定不输出这些敏感日志”，这里不再要求重新选择原 Q3 选项，只确认最小文档化边界。

## Q1: “不输出敏感日志”的文档边界如何写入？

1. 严格最小暴露。敏感字段至少包括 `password`、`uuid`、auth header、token-like query 参数、URL credential、TLS private/secret-like 字段；日志和错误 `msg`/fields 不输出原值；错误只允许输出 config path、字段名、错误类别和安全摘要；不提供 debug dump 绕过。  
   影响：最安全，文档和验收口径清晰；排障需要依赖 config path 和错误类别。  
   推荐项：是。  
   是否阻塞推进：是，阻塞 logging/error docs 完整更新。

2. 严格最小暴露，但允许显式本地 debug dump。默认不输出敏感原值；只有用户显式启用本地 debug dump 时才可输出完整配置，且必须标记危险、默认关闭、不得进入常规日志。  
   影响：排障能力更强，但需要额外配置、权限和安全提示。  
   推荐项：否。  
   是否阻塞推进：是。

3. 只写原则，不列字段清单。docs 只写“代码不得输出敏感日志”，具体敏感字段和摘要策略由实现时判断。  
   影响：最省文档工作，但后续实现和验收仍会存在解释空间。  
   推荐项：否。  
   是否阻塞推进：是。
