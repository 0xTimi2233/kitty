Role: PM

Goal:
审计用户在 `docs/` 下定义的项目初版需求，并结合 `crates/acl/` 已定义的配置输入 schema，识别正式 planning 前必须澄清或修正的需求缺口、冲突、假设和决策项。不要创建正式 run，不要更新 `codexspec/vision.md` 或 `codexspec/roadmap.md`。

Allowed input paths:
- `.codex/prompts/subagent-contract.md`
- `.codex/prompts/glossary.md`
- `.codex/prompts/report-contract.md`
- `codexspec/vision.md`
- `codexspec/roadmap.md`
- `docs/`
- `crates/acl/`

Input scope and reference expansion:
- `docs/` 是需求来源 scope，可按需读取其中的需求、设计草案、示例和引用文档。
- `crates/acl/` 是 schema/source-of-truth scope，可按需读取 schema、serde、默认值、normalize 及相关测试。
- 若 `docs/` 或 `crates/acl/` 明确引用 repo 内其他路径，可读取被引用的最小文件并在报告 `Inputs read` 中列出。
- 不要为了补全实现细节而扩展到整个 `crates/`；如需要更大代码上下文，返回 `needs-context` 并说明具体路径和原因。

Project rules:
- 工作流产物、自然语言正文使用简体中文。
- 路径、命令和专业名词保持英文。
- 这是 preflight audit，不是 implementation、architecture design、test planning 或 formal commit planning。
- 保留用户给出的目录 scope，不要把它们改写成完整文件清单作为 dispatch 输入；在 PM 报告中列出实际读取的文件。
- 需求审计应优先识别会阻塞后续 planning/design/execute 的问题，而不是展开实现方案。

Expected report path:
- `codexspec/runtime/preflight/preflight-20260427-225609/brief.md`

Required audit outputs:
- `sources.md`: 列出实际读取的需求源、schema 源及每个源的覆盖结论。
- `requirement-map.md`: 将抽取到的需求按功能需求、配置/schema 需求、运行时行为、性能/可靠性、安全/权限、运维/可观测性、测试/验收进行归类；每条需求必须包含来源、清晰度、是否被 `crates/acl/` schema 支撑、后续 planning 影响。
- `blocker-ledger.md`: 记录 planning 阻塞点，包含冲突、缺失范围、不可验收表述、schema 与需求不一致、跨模块边界不清、默认行为不清。
- `assumptions.md`: 记录可以暂时采用的假设和不应默认采用的假设，并说明风险。
- `decisions/queue.md`: 汇总需要用户确认的决策项，标明优先级、是否阻塞、关联需求。
- `decisions/batches/batch-001.md`: 给出第一批建议交给用户回答的问题。每个问题提供 2-4 个编号选项、影响和推荐项。
- `brief.md`: 给出审计摘要、ready-for-plan 状态、建议的下一步 `$plan` track，以及如果还不能进入 commit planning，需要先解决的最小决策集合。

Decision format:
- 只有确实需要用户或 PM 边界外决策时才返回 `Decision Request`。
- 决策项必须包含 2-4 个编号选项、影响、推荐项，以及是否阻塞推进。

Stop condition:
- 完成上述 preflight 产物并返回标准报告；或在缺少必要输入路径、权限或上下文时返回 `needs-context`。
